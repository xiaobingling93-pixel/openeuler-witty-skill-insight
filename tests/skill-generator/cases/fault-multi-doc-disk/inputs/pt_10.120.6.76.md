# 故障案例（问题单）

标题：硬盘物理介质损坏导致/dev/sdw磁盘I/O错误和XFS文件系统崩溃

环境：Linux / WUH721816ALE6L4 16TB硬盘 / XFS文件系统 / 影响范围：至少74个磁盘存在SMART问题，/dev/sdw最严重（必现）

【触发条件】
报告未提及具体触发操作，推测为磁盘在运行过程中逐渐出现坏道，从2026-02-22开始持续劣化，最终在2026-03-01 02:37:42爆发为系统性故障。

【现象】
2026-02-22 03:58:18开始，/dev/sdw磁盘首次报告8个当前不可读扇区，随后每小时报告一次。
2026-03-01 02:37:42出现mpt3sas_cm0控制器错误和批量I/O读取错误。
2026-03-01 02:38:13出现XFS日志I/O错误，文件系统开始关闭。

报错位置：系统日志（kernel日志）、SMART监控日志
报错字符串原文：
```
Mar  1 02:37:42 [mpt3sas_cm0] controller error (code 0x31110610)
Mar  1 02:38:13 XFS (sdw): log I/O error -5
```

【排查过程】
从报告中还原排查链路，每一步写清楚四个动作：

1. **observation**: 2026-02-22 03:58:18，/dev/sdw磁盘首次报告8个当前不可读扇区
   **thought**: SMART监控告警是硬件故障的早期预警信号，需要关注该设备的健康状态
   **action**: 查看SMART监控日志
   **finding**: 发现/dev/sdw（WUH721816ALE6L4 16TB硬盘）持续报告8个待处理扇区，且至少74个磁盘存在Current_Pending_Sector问题

2. **observation**: 2026-03-01 02:37:42，系统日志首次出现mpt3sas_cm0控制器错误
   **thought**: SMART告警早于系统I/O错误约7天，符合硬件故障特征，需要确定故障零点
   **action**: 分析系统日志时序，定位首次严重错误时间点
   **finding**: 确定故障零点T0为2026-03-01 02:37:42，出现控制器错误code 0x31110610和批量I/O错误

3. **observation**: 2026-03-01 02:38:13，出现XFS文件系统日志I/O错误
   **thought**: 硬件I/O错误已经影响到了文件系统层，需要确认文件系统损坏程度
   **action**: 查看文件系统层日志
   **finding**: XFS日志I/O错误检测，错误码为-5，文件系统强制关闭

4. **observation**: 多个磁盘出现I/O错误
   **thought**: 需要评估故障影响范围，确认是否只有单个磁盘问题
   **action**: 统计I/O错误分布
   **finding**: sdah 66次、sdad 46次、sdac 33次、sdw 12次，其他8个设备也有不同程度错误，说明系统稳定性受损

5. **observation**: 需要确认故障设备的物理位置
   **thought**: 为了更换故障硬件，需要确定设备对应的物理槽位
   **action**: 查询设备映射关系
   **finding**: /dev/sdw对应SCSI ID 0:0:22:0，SAS地址0x578aa82c42b7d015，具体物理槽位需要进一步确认

【根因】
现象：/dev/sdw设备出现持续8个待处理扇区，最终引发SAS控制器传输中断和XFS文件系统崩溃，多个其他磁盘也出现I/O错误
原因：/dev/sdw磁盘物理介质损坏。完整因果链：磁盘物理介质损坏 → 出现坏道（8个待处理扇区） → SAS控制器传输中断（code 0x31110610） → 块设备I/O错误 → XFS文件系统崩溃

【处置】
临时解（治标）：
- 立即更换/dev/sdw物理磁盘
- 尝试从备份恢复/dev/sdw上的数据
- 对受影响的XFS文件系统进行修复检查

根治解（治本）：
- 全面检查其他73个有SMART告警的磁盘，必要时更换
- 检查mpt3sas_cm0 SAS控制器健康状况
- 建立更早的磁盘健康度预警机制
- 考虑RAID配置提高数据可靠性
- 建立定期磁盘健康检查计划
