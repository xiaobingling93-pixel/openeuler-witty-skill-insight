# 故障案例（问题单）

标题：硬盘物理介质损坏导致/dev/sdi设备I/O读写失败

环境：Linux / WDC WUH721414AL 14TB硬盘 / XFS文件系统 / 影响范围：8号槽位硬盘（必现）

【触发条件】
报告未提及具体触发操作，推测为磁盘在正常运行过程中逐渐出现坏道，当业务访问到损坏扇区时触发I/O错误。

【现象】
在系统日志中首次出现严重I/O错误：`blk_update_request: I/O error, dev sdi`
报错位置：系统日志（kernel日志）
报错字符串原文：
```
Mar  5 14:11:36 shgh01-kss-yz112017032 kernel: [7353838.620832] blk_update_request: I/O error, dev sdi, sector 19327755344 op 0x0:(READ) flags 0x80700 phys_seg 4 prio class 0
Mar  5 14:11:36 shgh01-kss-yz112017032 kernel: [7353838.621010] blk_update_request: I/O error, dev sdi, sector 7541000824 op 0x0:(READ) flags 0x80700 phys_seg 95 prio class 0
```

【排查过程】
从报告中还原排查链路，每一步写清楚四个动作：

1. **observation**: 监控层持续报告/dev/sdi有240个不可读扇区
   **thought**: SMART监控告警是硬件故障的早期预警信号，需要关注该设备的健康状态
   **action**: 查看SMART监控日志
   **finding**: 发现/dev/sdi（WDC WUH721414AL 14TB硬盘）持续报告240个不可读扇区，且多个磁盘有重映射扇区计数和命令超时

2. **observation**: 2026-03-05 14.11.36系统日志首次出现严重I/O错误
   **thought**: SMART告警早于系统I/O错误，符合硬件故障特征，需要确定故障零点
   **action**: 分析系统日志时序，定位首次严重错误时间点
   **finding**: 确定故障零点T0为2026-03-05 14:11:36，出现blk_update_request I/O错误

3. **observation**: T0+28秒出现XFS文件系统元数据I/O错误
   **thought**: 硬件I/O错误已经影响到了文件系统层，需要确认文件系统损坏程度
   **action**: 查看文件系统层日志
   **finding**: XFS元数据读取失败，错误码为5（I/O错误）
   ```
   Mar  5 14:12:04 shgh01-kss-yz112017032 kernel: [7353866.621832] XFS (sdi): metadata I/O error in "xfs_trans_read_buf_map" at daddr 0x100002570 len 32 error 5
   ```

4. **observation**: 需要确认故障设备的物理位置
   **thought**: 为了更换故障硬件，需要确定设备对应的物理槽位
   **action**: 查询设备映射关系
   **finding**: /dev/sdi对应SCSI ID [0:0:8:0]，物理路径为pci-0000:65:00.0-sas-0x55098b8d3b9d3007-lun-0，推断物理位置为8号磁盘槽位

【根因】
现象：/dev/sdi设备出现大量坏道，导致数据读写失败，XFS文件系统元数据读取失败
原因：硬件磁盘物理介质损坏。完整因果链：磁盘物理介质损坏 → 出现坏道（240个不可读扇区） → I/O读写操作失败（error 5） → 文件系统元数据损坏

【处置】
临时解（治标）：
- 立即隔离故障磁盘：将/dev/sdi从业务中移除
- 如有重要数据，立即尝试备份
- 监控RAID组中其他磁盘的健康状态

根治解（治本）：
- 物理更换8号槽位的硬盘（WDC WUH721414AL 14TB硬盘）
- 如有RAID配置，启动重建过程
- 定期检查所有磁盘的SMART状态
- 设置更严格的磁盘健康告警阈值

命令填写规则：
- 查看SMART状态：`smartctl -a /dev/sdi`（报告未出现，根据经验补充，未验证）
