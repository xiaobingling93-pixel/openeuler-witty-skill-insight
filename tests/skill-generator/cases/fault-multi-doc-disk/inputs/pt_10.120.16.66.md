# 故障案例（问题单）

标题：硬盘物理介质损坏导致/dev/sdai设备不可恢复的读取错误

环境：Linux / ST14000NM0018-2H 14TB HDD / 报告未提及平台版本 / 影响范围：第35个物理磁盘槽位（必现）

【触发条件】
报告未提及具体触发操作，推测为磁盘在运行过程中出现介质损坏，从3月2日开始出现间歇性I/O错误，3月3日23:03达到高峰。

【现象】
2026-03-03 23:03:00，系统日志出现物理磁盘介质错误和不可恢复的读取错误。

报错位置：系统日志（kernel日志）
报错字符串原文：
```
```
Mar  3 23:03:00 bjwg01-ks3-bj-standard120016066 kernel: [13853375.445432] sd 0:0:34:0: [sdai] tag#5104 Sense Key : Medium Error [current] [descriptor]
Mar  3 23:03:00 bjwg01-ks3-bj-standard120016066 kernel: [13853375.445624] sd 0:0:34:0: [sdai] tag#5104 Add. Sense: Unrecovered read error
Mar  3 23:03:00 bjwg01-ks3-bj-standard120016066 kernel: [13853375.446314] blk_update_request: critical medium error, dev sdai, sector 22969376632 op 0x0:(READ) flags 0x80700 phys_seg 11 prio class 0
```

【排查过程】
从报告中还原排查链路，每一步写清楚四个动作：

1. **observation**: 2026-03-03 23:03:00，系统日志出现物理磁盘介质错误
   **thought**: 物理磁盘介质错误是硬件故障的底层信号，需要确定故障设备
   **action**: 分析系统日志中的SCSI错误信息
   **finding**: 确定故障设备为/dev/sdai，SCSI地址为0:0:34:0，错误类型为Medium Error（介质错误）和Unrecovered read error（不可恢复的读取错误）

2. **observation**: 系统日志出现critical medium error
   **thought**: 关键介质错误说明磁盘物理损坏严重，需要确认故障零点
   **action**: 分析日志时序，定位首次严重错误时间点
   **finding**: 确定故障零点T0为2026-03-03 23:03:00，涉及扇区22969376632的读取操作失败

3. **observation**: 需要确认故障设备的物理位置
   **thought**: 为了更换故障硬件，需要确定设备对应的物理槽位
   **action**: 查询设备映射关系
   **finding**: /dev/sdai对应SCSI地址0:0:34:0，驱动器ID 34（RAID配置中的37:34），推断物理位置为第35个物理磁盘槽位（从0开始计数）

4. **observation**: 需要确认是否只有单个磁盘问题
   **thought**: 需要评估故障影响范围，检查其他磁盘的健康状态
   **action**: 查看SMART日志和RAID状态
   **finding**: SMART日志显示多个磁盘存在重映射扇区（Reallocated_Sector_Ct）和命令超时（Command_Timeout）警告，RAID状态显示所有物理驱动器为UGood状态，但SMART指标显示潜在问题

【根因】
现象：/dev/sdari设备出现物理磁盘介质错误，导致不可恢复的读取错误和关键介质错误，扇区22969376632读取失败
原因：物理磁盘硬件故障，具体为磁盘介质（Medium）损坏。完整因果链：磁盘介质损坏 → SCSI驱动报告介质错误 → 内核报告critical medium error → 数据读取操作失败

【处置】
临时解（治标）：
- 立即更换故障物理磁盘（驱动器ID 34）
- 如有RAID保护，启动重建过程
- 监控其他磁盘的SMART状态，预防连锁故障

根治解（治本）：
- 更换第35个槽位的硬盘（ST14000NM0018-2H 14TB HDD）
- 检查其他磁盘的健康状态，必要时更换
- 加强磁盘健康监控，设置更严格的告警阈值
- 确保重要数据有完整备份
