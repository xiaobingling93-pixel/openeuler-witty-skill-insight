## Issue Description

## 1、背景与价值
目前平台已初步构建了“具体案例 - 故障模式抽取 - Skill 生成”的链路，利用故障模式作为中间抽象层，有效缓解了从原始数据到 Skill 的泛化性低的问题。
然而，当前的自动化流程仍存在以下局限：
- 黑盒生成： 用户无法介入生成过程，导致生成的 Skill 可能不符合实际运维直觉。
- 知识孤岛： 生成过程仅依赖当前故障案例，无法有效引入运维专家的通用知识或历史经验。
- 静态管理： 故障模式库缺乏更新机制，无法随业务演进自动迭代。

通过引入用户交互、经验融合及故障模式管理机制，可以显著提升 Skill 生成的准确性和可解释性，最终实现 openEuler 真实场景下的高效赋能。

## 2、需求详情
### 增加用户互动与反馈回路：
- 在故障案例生成后、故障模式抽取后和 Skill 生成后增加“用户确认&修改”环节。
- 也提供无交互模式，不需要用户参与即可自动生成skill。

### 多源知识融合注入：
- 扩展输入接口，支持在生成 Skill 时同步输入“通用运维经验”或“特定领域知识”。用户交互时，需要支持用户直接输入的文本内容，也可以指定文件路径，作为经验融合的内容。
- 允许支持“故障模式 + 通用经验”的联合推理生成 skill。

## 3、验收标准
- 交互链路完备性：用户能够在界面上查看到中间态的故障案例、故障模式以及最终的skill，并能手动输入修改信息参与 Skill 的生成过程。skill生成后，需要用户确认。
- 知识多源性：在生成 Skill 时支持用户交互补充经验或者特定领域知识，共同生成skill
- 提供无交互模式，不需要用户参与即可自动生成skill。
