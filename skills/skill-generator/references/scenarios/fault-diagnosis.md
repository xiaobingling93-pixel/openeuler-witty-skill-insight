# 故障诊断场景：Skill 生成工作流

> 本文件由 `skill-generator` 的 SKILL.md 加载。适用于：故障排查、异常诊断、告警分析等场景。
>
> 前提：若用户提供了文档，SKILL.md 路由层已完成文本提取。本模块接收的输入为用户直接提供的文本/数据，以及已提取的文档内容。

本工作流的目标：**将用户输入收敛为一份结构化的故障模式列表，并据此生成符合规范的 Skill 包。**

**流程总览**：Step 0（构建故障模式列表）→ Step 1（质量扫描）→ Step 2（搜索补全）→ Step 3（预填充审核）→ Step 4（生成）

Step 0 的工作量取决于输入的完备程度。输入为完整的故障模式列表时，Step 0 直接透传；输入为领域描述或非结构化案例时，Step 0 需完成模式生成或归纳。

---

## Step 0：构建故障模式列表

### 输入分类与处理

根据当前可用信息（用户直接输入 + 文档提取内容）选择对应处理路径：

| 可用输入 | 处理路径 |
|---|---|
| 结构化故障模式列表（用户直接提供，或文档中包含的已整理清单） | 直接进入 Step 1 |
| 单一故障描述（如 "SSH 登录失败"） | 通过搜索和领域知识扩展为多个故障模式 |
| 领域描述（如 "Kafka 排查"） | 通过搜索和领域知识生成故障模式列表，经用户确认后继续 |
| 故障案例/问题单/排障记录（用户直接提供，或从文档中提取） | 提取案例并归纳为故障模式 |
| 上述输入的任意组合 | 分别处理后整合（去重、合并、补充） |

> 注意：同一份文档可能同时包含已整理的故障模式列表和零散的案例记录。列表部分直接采用，案例部分按归纳流程处理后整合。

### 从描述或领域生成故障模式

当用户仅提供故障现象描述或领域名称时，按以下步骤处理：

1. **搜索调研**：检索该领域/故障的常见子场景与典型排查路径
2. **生成列表**：基于搜索结果与领域知识，列出主要故障模式
3. **用户确认**：将列表呈现给用户，说明每个模式的含义与覆盖范围，由用户增删调整

单一故障描述与领域描述的区别在于粒度（前者从单一故障扩展子模式，后者需系统性梳理整个领域），处理流程一致。

> 注意：用户可能不熟悉"故障模式"的概念。呈现列表时采用易理解的格式：每个模式提供名称 + 一句话说明其典型触发条件。

### 从案例归纳故障模式

当输入包含具体的故障事件记录（问题单、排障日志、案例报告等）时，先提取再归纳。

**提取故障案例**——从内容中识别每个独立的故障事件，提炼以下字段：
- `title`：`[组件] + [根因摘要] + [现象摘要]`
- `timeline`：事件时间线
- `trigger_event`：触发事件
- `evidences`：关键证据
- `commands`：原始命令

**归纳故障模式**——将案例抽象为泛化的故障模式。核心原则：**提取底层机制，而非堆砌具体细节**。每个故障模式应包含：
- `pattern_name`：泛化名称（不得包含真实 IP/主机名等环境特定信息）
- `summary`：诊断手段、问题分类、宏观因果描述
- `fault_mechanism`：技术原理层面的故障成因解释
- `diagnosis_steps`：结构化排查步骤——`action` → `where` → `command`（使用 `{{PLACEHOLDER}}` 替代环境参数）→ `expected_output` → `critical`
- `variation_vectors`：该故障在不同环境/参数下重现时的差异维度

**多案例/多文档处理**：逐份处理时，先与已归纳的 Pattern 比对。属于同一故障域则更新已有 Pattern 的 variation_vectors；超出已有范围则拆为新 Pattern。

### 混合输入整合

当存在多种来源的故障模式时（如用户提供了列表，同时附带了问题单），按以下方式整合：
1. 各来源分别处理，各自产出故障模式
2. 交叉比对合并：
   - 与已有列表重叠的模式 → 用新来源的案例数据补充其检测方法、恢复方法、variation_vectors
   - 新来源中出现的新模式 → 作为候选项追加，在 Step 3 中交由用户确认是否纳入
3. 整合后的完整列表作为 Step 1 的输入

## Step 1：解析与质量扫描

接收 Step 0 产出的故障模式列表（或用户直接提供的列表），执行质量扫描。**禁止跳过此步直接进入生成。**

### 解析

无论输入格式为表格、YAML、JSON 或自由文本，统一提取以下字段：
- 故障域（一级/二级分类）
- 每个故障模式的：名称、描述、影响范围、检测方法（可能缺失）、恢复方法（可能缺失）

### 质量扫描

检查以下 5 类问题（详细规则见 `templates/fault-diagnosis/quality_scan.md`）：

1. **重复**：相同名称出现多次 → 分析差异，准备合并方案
2. **层级关系**：多个模式间存在隐含的排查顺序依赖（如"先查 A 再查 B/C/D"）→ 识别并记录
3. **缺失字段**：检测方法或恢复方法为空 → 基于领域知识预填充
4. **内部工具引用**：出现无法识别的工具名称或缩写 → 推测候选并准备确认
5. **模糊描述**：如"需要人为介入"等笼统表述 → 细化为具体操作步骤

## Step 2：搜索补全

逐一评估每个故障模式：**对其检测方法和恢复方法的置信度是否足够？**

满足以下任一条件时触发搜索：
1. 检测方法和恢复方法均缺失
2. 对具体命令或参数存在不确定性
3. 涉及特定版本或发行版差异
4. 故障描述模糊，无法直接映射到具体排查命令

每个故障模式最多执行 2 次搜索。搜索策略：
```
# 检测方法缺失
搜索: "{故障现象} linux troubleshoot command"

# 恢复方法缺失
搜索: "{故障现象} linux fix resolve"

# 命令参数不确定
搜索: "{具体命令} usage example"
```

> **依赖说明**：搜索能力取决于外层 Agent 框架是否配置了搜索工具。若搜索不可用，基于已有知识尽力补全，并在 Step 3 的审核环节中向用户标注哪些内容未经搜索验证。

## Step 3：预填充审核

将质量扫描发现的问题和搜索补全的结果转化为**选择题**呈现给用户。当且仅当你实在无法作出判断时，才需要询问用户。核心原则：**用户无需从空白开始填写，只需确认或修正预填充内容。**

三种题型（总数 ≤ 8 题；超出时对高置信度问题自动采纳推荐选项）：

**单选题**（结构性决策）：
```
Q: "用户态内存泄漏"出现 3 次，应如何处理？
(A) 合并为 1 个故障模式 ← 推荐
(B) 保持 3 个独立故障模式
```

**确认题**（高置信度推断）：
```
Q: 排查顺序设定为"先用户态后内核态"，是否确认？[Y/N] ← 推荐 Y
```

**填空预填充题**（缺失内容补全）：
```
Q: "total 内存不足"缺少检测方法，建议使用：
  dmidecode -t memory | grep "Size:"
  grep MemTotal /proc/meminfo
(A) 采纳 ← 推荐
(B) 修改为：______
```

若用户表达"自动"、"跳过"或 `--auto`，全部采纳推荐选项，跳过交互环节。

## Step 4：生成 Skill

基于审核确认后的数据生成完整 Skill 目录。

> 前提：`references/skill-template.md` 已在流程起始阶段加载，所有产物须符合其规范。

### 产出目录结构

```
{skill_name}/
  SKILL.md                    # 排查主文档（符合 skill-template.md 规范）
  scripts/
    collect.sh                # 信息采集脚本（自包含，无外部依赖）
    check_{fault_mode}.sh     # 每个故障模式对应一个排查脚本（自包含）
  references/
    {fault_mode}.md           # 每个故障模式的子场景分析与排查建议
    failure_cases.yaml         # 仅案例归纳场景：原始案例存档
    pattern_detail.md          # 仅案例归纳场景：归纳细节与 variation vectors
```

详细的产出结构和脚本生成规则见 `templates/fault-diagnosis/output_structure.md`。
排查决策树生成规则见 `templates/fault-diagnosis/triage_prompt.md`。

### 4.1 SKILL.md 生成规范

生成的 SKILL.md 是最终 Skill 被 Agent 加载后的核心执行文件。

排查型 Skill 的关键设计原则：**Agent 先识别故障场景，再进入对应故障模式的排查流程**。禁止让 Agent 顺序执行所有脚本。

**核心机制**：
1. Agent 综合用户描述的现象和采集到的环境信息，自主判断最可能的故障模式
2. 确定故障模式后，加载该模式的 reference 文件（`references/{fault_mode}.md`），了解子场景和排查建议
3. 根据 reference 中的子场景进一步定位，运行对应的排查脚本验证
4. 未命中时回到模式识别环节，尝试下一个候选模式

生成的 SKILL.md 须包含 YAML frontmatter（规范见 `references/skill-template.md`），主体结构如下：

```markdown
# {故障域}排查

> 适用: {环境} | 版本: v1.0

## 概述 (Overview)
{故障域简介，2-3 行}

## 核心指令 (Core Instructions)

### Step 1: 信息采集

运行基础信息采集，获取当前环境状态：
```bash
sudo bash scripts/collect.sh | tee /tmp/diag_collect.json
```

### Step 2: 故障模式识别

综合以下信息，判断当前最可能的故障模式：
- 用户描述的故障现象和上下文
- Step 1 采集到的系统状态信息
- 各故障模式的典型特征（见下表）

| # | 故障模式 | 典型现象 | 关键判据 |
|---|---|---|---|
| 1 | {模式名称} | {一句话描述典型表现} | {区分该模式的关键信号} |
| 2 | {模式名称} | {一句话描述典型表现} | {区分该模式的关键信号} |
| ... | ... | ... | ... |

> 识别原则：
> - 根据现象与关键判据的匹配度选择最可能的模式，不要顺序遍历
> - 若多个模式的特征同时出现，优先选择底层/根因侧的模式
> - 若信息不足以判断，向用户追问关键区分信息

### Step 3: 场景细化与排查

确定故障模式后：

1. **加载场景详情**：读取 `references/{selected_mode}.md`，了解该模式下的子场景分类和排查建议
2. **子场景定位**：根据 reference 中的子场景描述，结合已有信息进一步缩小范围
3. **执行验证**：运行对应的排查脚本确认：
```bash
sudo bash scripts/check_{selected_mode}.sh
```
4. **结果解读**：
   - **命中 (HIT)** → 结合 reference 中的排查建议执行修复
   - **未命中 (MISS)** → 返回 Step 2，根据脚本输出的排除信息选择下一个候选模式

> **追问原则**：命中故障后继续追问以下三个问题：
> 1. 何时开始？— 检查相关日志时间线
> 2. 何事触发？— 检查近期变更记录
> 3. 影响范围？— 排查跨层传播

### 兜底

所有故障模式均未命中时：
1. `bash scripts/collect.sh --full`
2. 上报至 {升级路径}

### 诊断结论模板
```
故障根因:
故障组件:
故障时间: [首次] → [确认]
故障链:   [T1]A → [T2]B → [T3]C
已排除:   {项}: {依据}
定位依据: {关键证据}
修复建议: 临时 / 根因 / 预防
```

## 参考文件说明
- `scripts/collect.sh`: 系统信息采集（自包含脚本）
- `scripts/check_*.sh`: 各故障模式排查脚本（自包含，可独立执行）
- `references/{fault_mode}.md`: 各故障模式的子场景分析与排查建议
```

**控制原则**：
- 每个故障模式在 Step 2 表格中占一行，表格侧重**区分特征**以帮助 Agent 识别
- 详细的子场景分析和排查建议放在 `references/{fault_mode}.md` 中
- SKILL.md 总行数控制在 500 行以内

### 4.2 排查脚本生成规范

所有脚本必须**自包含**——不依赖外部文件，可单独执行。所需的辅助函数直接内联在脚本头部。

可用的辅助函数定义在 `templates/fault-diagnosis/_lib.sh` 中，生成时按需选取内联（不要全量复制，只取脚本实际调用的函数）。

**collect.sh**：
```bash
#!/bin/bash
set -uo pipefail

# --- 内联辅助函数 ---
DIAG_RESULTS=()
record() {
    local check="$1" status="$2" detail="$3" ts
    ts=$(date -Iseconds 2>/dev/null || date "+%Y-%m-%dT%H:%M:%S")
    detail=$(printf '%s' "$detail" | head -20 | tr '\n' '|' | tr -d '\r\t')
    detail=$(printf '%s' "$detail" | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    check=$(printf '%s' "$check" | tr -d '\r\t' | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    ts=$(printf '%s' "$ts" | tr -d '\r\t' | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    DIAG_RESULTS+=("{\"check\":\"$check\",\"status\":\"$status\",\"detail\":\"$detail\",\"time\":\"$ts\"}")
}
dump_json() {
    echo "["; local first=true
    for r in "${DIAG_RESULTS[@]}"; do $first && first=false || echo ","; echo "  $r"; done
    echo "]"
}
# --- 辅助函数结束 ---

echo "=== 信息采集 $(date) ==="
record "sysinfo" "INFO" "$(uname -a; cat /etc/os-release 2>/dev/null)"
record "uptime" "INFO" "$(uptime)"
record "memory" "INFO" "$(free -h)"
record "disk" "INFO" "$(df -h)"
record "top_procs" "INFO" "$(ps aux --sort=-%mem | head -10)"
# {域特定采集项}
dump_json
```

**check_{fault_mode}.sh**——每个故障模式对应一个自包含脚本：
```bash
#!/bin/bash
set -uo pipefail

# --- 内联辅助函数（按需选取） ---
DIAG_RESULTS=()
run_cmd() {
    local desc="$1"; shift; local output rc
    output=$(timeout 10 "$@" 2>&1); rc=$?; echo "$output"; return $rc
}
record() {
    local check="$1" status="$2" detail="$3" ts
    ts=$(date -Iseconds 2>/dev/null || date "+%Y-%m-%dT%H:%M:%S")
    detail=$(printf '%s' "$detail" | head -20 | tr '\n' '|' | tr -d '\r\t')
    detail=$(printf '%s' "$detail" | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    check=$(printf '%s' "$check" | tr -d '\r\t' | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    ts=$(printf '%s' "$ts" | tr -d '\r\t' | sed -e 's/\\/\\\\/g' -e 's/"/\\"/g')
    DIAG_RESULTS+=("{\"check\":\"$check\",\"status\":\"$status\",\"detail\":\"$detail\",\"time\":\"$ts\"}")
}
hit()  { record "$1" "HIT"  "$2"; }
miss() { record "$1" "MISS" "$2"; }
dump_json() {
    echo "["; local first=true
    for r in "${DIAG_RESULTS[@]}"; do $first && first=false || echo ","; echo "  $r"; done
    echo "]"
}
# --- 辅助函数结束 ---

echo "=== {故障模式名}排查 $(date) ==="

# Check 1: {检查项描述}
output=$(run_cmd "{描述}" {命令})
if {判断条件}; then
    hit "{检查名}" "$output"
else
    miss "{检查名}" "$output"
fi

# Check 2: {检查项描述}（同一故障模式下的关联检查）
# ...

dump_json
```

**脚本约束**：
- 使用 bash 编写，兼容 CentOS/RHEL/Ubuntu
- **自包含**：每个脚本独立可执行，不 `source` 外部文件
- 辅助函数从 `templates/fault-diagnosis/_lib.sh` 按需内联，仅包含脚本实际使用的函数
- 仅使用系统自带命令，不依赖第三方工具
- 每个脚本对应一个故障模式，内部包含 2-4 个关联检查步骤
- 脚本命名须反映故障模式语义：如 `check_memory_leak.sh`、`check_disk_full.sh`，禁止使用 `check_1.sh` 等无意义编号

### 4.3 故障模式 Reference 生成规范

每个故障模式**必须**生成对应的 `references/{fault_mode}.md` 文件。此文件供 Agent 在识别故障模式后加载，用于子场景细化和排查指导。

**文件结构**：

```markdown
# {故障模式名称}

## 概述
{该故障模式的机制简述，1-2 行}

## 子场景

### 子场景 1: {子场景名称}
- **典型表现**: {该子场景特有的症状}
- **触发条件**: {导致该子场景的常见原因}
- **排查建议**:
  1. {具体排查步骤与命令}
  2. {进一步确认方法}
- **修复建议**: {临时缓解 / 根因修复}

### 子场景 2: {子场景名称}
...

## 关联故障
- 与 {其他故障模式} 的区分: {关键区分点}
- 可能的连锁故障: {该模式可能引发的下游问题}
```

**生成原则**：
- 每个故障模式拆分为 2-5 个子场景，子场景是该故障模式下的具体变体
- 子场景的排查建议须包含**具体命令**和**预期输出**，不得使用"检查..."等模糊表述
- 修复建议区分临时缓解和根因修复
- 关联故障部分帮助 Agent 在当前模式未命中时快速切换方向
- 案例归纳场景中的 variation vectors 应体现为子场景的差异维度

**与脚本的关系**：
- `check_{fault_mode}.sh` 负责**自动化检测**（跑命令、判断 hit/miss）
- `references/{fault_mode}.md` 负责**场景理解与决策指导**（子场景区分、排查建议、修复策略）
- 二者互补：Agent 先读 reference 理解场景，再跑脚本验证，最后结合 reference 中的建议给出结论

## 常见判断

**故障模式拆分粒度？**
- 每个故障模式应为一个可独立排查的故障假设
- 粒度过粗（如"内存问题"）→ 应拆分为"用户态内存泄漏"、"内核态内存泄漏"、"OOM Killer 触发"等
- 粒度过细（如"特定进程在特定条件下的内存泄漏"）→ 应合并为更通用的模式

**用户仅提供一句话故障描述？**

**用户同时提供了故障模式列表和案例/问题单？**
- 先处理案例内容（判断是已整理的列表还是需要归纳的案例记录），按对应方式处理后与已有列表整合。重叠部分补充细节，新增部分在审核环节交由用户决定是否纳入。
