# 通用场景：Skill 生成工作流

> 本文件由 `skill-generator` 的 SKILL.md 加载。适用于：根据用户描述的主题或需求，由 Agent 直接驱动生成 Skill，无需文档输入或故障模式列表。
>
> **核心路径**：Agent 驱动，不依赖 Python 脚本。通过理解需求、搜集知识、设计结构，直接生成符合规范的 Skill 目录。

---

## 适用范围

| 适用场景 | 示例 |
|---|---|
| 操作手册类 | K8s 部署 Checklist、数据库备份恢复操作 |
| 工具使用类 | Terraform 项目初始化、Git 工作流 |
| 知识库/最佳实践类 | React 最佳实践、安全合规检查表 |
| 文档创作类 | 技术博客写作、API 文档生成 |
| 流程指导类 | Code Review Checklist、上线流程 |

> 如果用户的需求是**从已有文档提取 Skill**（PDF/MD/URL），请转至 `references/scenarios/doc-pipeline.md`。

---

## 工作流程

整个生成分五步：**理解需求 → 搜集知识 → 设计结构 → 确认大纲 → 生成产出**。不要跳步。

### Step 1：需求理解

拿到用户描述后，先澄清四个维度（用已有信息推断，缺失的才询问）：

| 维度 | 问题 | 示例 |
|---|---|---|
| **主题** | 这个 Skill 关于什么？ | "Git 代码提交规范" |
| **受众** | 谁会触发这个 Skill？ | "团队开发者（不熟悉规范的新成员）" |
| **核心动作** | Skill 要完成什么任务？ | "指导写出合规的 commit message" |
| **输出期望** | 用户期望什么产出？ | "有步骤说明 + 示例 + 检查清单" |

**推断规则**：
- 如果用户只给了主题（如"帮我做个 Git 工作流的 Skill"），可以合理推断受众和核心动作，**不需要额外问**
- 如果用户的描述本身很完整，直接进入 Step 2
- 只有当推断会引发明显歧义时，才用一句话主动询问

### Step 2：知识搜集

根据主题评估：**我对这个领域的最佳实践、核心步骤、常见工具有多大把握？**

**触发搜索的条件**（满足任一即搜索）：
1. 主题涉及特定版本/发行版/框架差异 → 搜索确认
2. 核心步骤中有不确定的命令或参数 → 搜索确认
3. 需要了解行业最佳实践/标准 → 搜索补全
4. 主题较新或小众，自身知识可能过时 → 搜索验证

每个知识点最多搜索 **2 次**。搜索策略：
```
# 了解最佳实践时
搜索: "{主题} best practices guide"

# 确认具体命令/参数时
搜索: "{命令或工具名} {具体用法} example"
```

搜索完成后，确认已掌握：核心步骤列表、关键工具/命令、常见错误或注意事项。

### Step 3：结构设计

根据主题和需求，选择合适的工作流模式（见 `references/skill-template.md`），并规划产出结构：

**工作流模式选择**：

| 场景特征 | 推荐模式 |
|---|---|
| 步骤有严格依赖顺序 | 顺序工作流 |
| 不同条件走不同路径 | 条件分支 |
| 需要执行-验证-修正循环 | 验证循环 |
| 多个互不依赖的子任务 | 并行执行 |

**产出结构决策**：

| 决策 | 规则 |
|---|---|
| 需要 `scripts/` 吗？ | 有需要执行的命令集合，且会被反复调用 → 要 |
| 需要 `references/` 吗？ | 有超过 5 行的背景知识、工具说明、示例集合 → 要 |
| SKILL.md 行数控制 | 核心指令控制在 30 行内，详细内容拆到 references/ |

规划 Skill 大纲（标题、步骤列表、脚本/文档规划）备 Step 4 确认。

### Step 4：确认大纲

向用户呈现 Skill 大纲，格式如下：

```
📋 Skill 生成大纲

名称: {skill-name}
主题: {一句话描述}
工作流模式: {顺序/条件分支/验证循环/并行}

核心步骤:
  Step 1: {步骤名} — {一句话说明}
  Step 2: {步骤名} — {一句话说明}
  ...

产出目录:
  SKILL.md         # 主文档（含步骤指令）
  scripts/         # （如有）列出脚本名和用途
  references/      # （如有）列出文档名和内容

---
(A) 确认，开始生成
(B) 修改: ______
```

如果用户说"确认"、"ok"、"直接生成"或"--auto"，直接进入 Step 5。

### Step 5：生成产出

> 注：此时 `references/skill-template.md` 已在 Step 2 中加载，所有产物需符合其标准。

按确认后的大纲，生成完整的 Skill 目录：

```
{skill_name}/
  SKILL.md                    # 主文档（符合 skill-template.md 规范）
  scripts/                    # 按需生成
    {action}.sh               # 可执行脚本（幂等、JSON 输出）
  references/                 # 按需生成
    {topic}.md                # 背景知识、工具说明、示例集合
```

#### 5.1 SKILL.md 生成规范

必须包含 YAML frontmatter。主体结构参考 `references/skill-template.md`。

**keyword 提取规则**：严格从 Skill 内容提取，包含：
- 工具名/技术栈（如 Git、Terraform、K8s）
- 核心操作动词（如"部署"、"提交"、"检查"）
- 适用环境（如适用）

#### 5.2 scripts/ 生成规范

- 只在用户明确需要自动化操作，或步骤中有可复用的命令集时生成
- 脚本必须**幂等**，输出 JSON 或结构化文本
- 不生成纯检查类脚本——简单的检查命令直接写在 SKILL.md 步骤里即可

#### 5.3 references/ 生成规范

- 超过 5 行的背景知识 → 拆到 `references/{topic}.md`
- 示例集合（多个示例）→ 拆到 references
- SKILL.md 正文只保留引用说明，不重复内容

---

## 常见判断

**要不要生成脚本？**
- 只有一两条命令 → 不要，直接写在步骤里
- 命令需要参数处理或条件判断 → 要
- 用户会反复执行同一套命令 → 要

**references/ 里放什么？**
- 工具的详细参数说明（超过 5 行）
- 示例代码或配置（多个）
- 背景理论（为什么这样做）

**步骤要多细？**
- 每个 Step 不超过 6 行
- 命令要可直接复制执行，不要用"检查...是否..."代替具体命令
- 复杂判断用条件分支格式，不要用自然语言描述

## 示例

**示例 A：操作手册（Git Commit 规范）**

1. 理解：主题=Git commit message 规范，受众=开发者，动作=指导写规范 commit
2. 搜集：搜索 "conventional commits specification"，确认格式规则
3. 结构：顺序工作流，无需脚本，references/ 放示例集合
4. 大纲确认：4个步骤（格式说明→类型选择→范围填写→完整示例）
5. 生成：SKILL.md + references/commit-examples.md

**示例 B：工具使用（Terraform 初始化）**

1. 理解：主题=Terraform 项目初始化，受众=基础设施工程师，动作=初始化新项目
2. 搜集：搜索 "terraform init best practices provider lock"，确认命令参数
3. 结构：顺序工作流 + 验证循环（init → validate → plan），需要 scripts/
4. 大纲确认：5个步骤，生成 scripts/init.sh
5. 生成：SKILL.md + scripts/init.sh + references/provider-config.md

**示例 C：用户只给一句话**

用户输入：> "做个 Code Review checklist 的 skill"

1. 理解：主题=Code Review，受众推断=开发者/reviewer，动作推断=提供系统化审查清单
2. 搜集：搜索 "code review checklist best practices"，归纳检查维度
3. 结构：顺序工作流，按维度拆步骤（代码质量→安全→测试→文档），references/ 放详细规则
4. 大纲确认：呈现给用户确认
5. 生成
