# Skill 标准输出规范

> 本文档是**所有场景共用的输出模板**。生成任何类型的 Skill 时，最终产物必须符合本规范。

---

## 1. 目录结构规范

```
{skill_name}/
  SKILL.md              # 必需：Skill 主文档
  scripts/              # 可选：可执行脚本
  references/           # 可选：参考文档和知识库
  templates/            # 可选：生成物模板（用于生成二次产物的场景）
```

---

## 2. SKILL.md 格式规范

### 2.1 YAML Frontmatter（必需）

```yaml
---
name: [skill-name]
# 规则：小写字母 + 数字 + 连字符，最多 64 字符，与目录名一致
# 示例：os-oom-diagnostics, ssh-login-failure-triage

description: >
  [详细描述，最多 1024 字符，不少于 100 字符]
# 重要：这是 Agent 激活 Skill 的唯一依据。必须包含：
#   1. 该 Skill 做什么
#   2. 何时触发（触发关键词/场景）
#   3. 略微"激进"——宁可多触发也不要漏触发

---
```

---

### 2.2 Skill 主体结构（必需遵循顺序）

```markdown
# [skill 名称]

## 概述 (Overview)
简要说明此 Skill 的用途、适用场景、核心价值。（3-5 行）

## 核心指令 (Core Instructions)
Skill 的主体执行部分。
- 每个步骤有明确的目标和命令
- 包含清晰的条件分支（如果...则...）
- 执行步骤引用 scripts/ 下的脚本（如有）
- 需要用户执行的修复操作，放到"建议"部分而非"执行"步骤

### 步骤 1：[步骤名称]
...

[选择合适的工作流模式，见下方]

## 参考文件说明
列出所有参考文件的路径，并简要说明内容。
```

---

### 2.3 工作流模式（按需选用）

**顺序工作流**：用于有依赖关系的线性步骤
```markdown
> 状态追踪：
- [ ] Step 1: ...
- [ ] Step 2: ...
```

**条件分支**：用于需要根据检测结果分路径的场景（故障诊断常用）
```markdown
**分支逻辑：**
- 场景 A（条件 X）→ 执行 ...
- 场景 B（条件 Y）→ 执行 ...
- ⚠️ 兜底：记录现象，上报
```

**验证循环**：用于需要执行-验证-修正的场景
```markdown
循环：检测 → 判断 → (失败则修复 → 重试) → 成功
```

**并行执行**：用于互不依赖的任务
```markdown
同时执行：
├── 任务 A
├── 任务 B
└── 任务 C
```

## 3. 规范约束

| 约束 | 规范值 |
|---|---|
| SKILL.md 最大行数 | ≤ 500 行 |
| description 字符数 | 100 ~ 1024 |
| name 格式 | kebab-case，≤ 64 字符 |
| 步骤中的命令 | 必须是可直接执行的具体命令，不得用"检查..."代替 |
| 修复/变更操作 | 不得在执行步骤中直接执行，放到"建议"中由用户决定 |
| 详细背景知识 | 超过 5 行的内容拆到 `references/`，正文只留引用 |

---

## 4. references/ 文件规范

- 所有文件使用 Markdown 格式（`.md`）
- 建议创建 `index.md` 作为目录索引
- 内容面向 Agent 的懒加载读取：回答"为什么"和"怎么判断"，不重复 SKILL.md 中的命令

---

## 5. scripts/ 文件规范

- 脚本必须**幂等**：多次执行不改变系统状态
- 默认只读操作；变更操作需显式声明
- 输出格式统一为 JSON 或结构化文本，便于 Agent 解析
